架构与路由

Glue2 是位于所有 HolaCloud 服务前面的中央反向代理。所有传入流量——无论是来自浏览器用户、API 客户端还是内部服务——在到达目的地之前都经过 Glue2。

反向代理

Glue2 作为第 7 层反向代理运行。它终止 TLS 连接，检查每个请求，验证调用者身份，然后将请求转发到相应的后端服务。没有外部流量直接到达 HolaCloud 服务；Glue2 是唯一的入口点。

1客户端 ──▶ Glue2（身份验证检查）──▶ 后端服务
2              │
3              ├──▶ InceptionDB
4              ├──▶ Lambda
5              ├──▶ Files
6              ├──▶ Config
7              ├──▶ KVNode
8              ├──▶ Scheduler
9              └──▶ Logs

基于虚拟主机的路由

Glue2 根据 

1Host
标头路由请求。每个 HolaCloud 项目被分配一个唯一的子域名（
1<项目>.hola.cloud
），Glue2 将该子域名映射到项目的后端服务。

内置平台服务使用保留子域名：

主机 服务 
1inceptiondb.hola.cloud
 InceptionDB 
1api.hola.cloud
 公共 API（Lambda、Files、Config 等） 
1console.hola.cloud
 HolaCloud 控制台 
1auth.hola.cloud
 身份验证 / 会话服务 
1logs.hola.cloud
 InstantLogs

当请求到达 

1my-project.hola.cloud
时，Glue2 查找具有该别名的项目，解析其主后端服务，并转发请求。

代理流程

  1. 客户端向 
    1<项目>.hola.cloud
     发送 HTTP 请求。
  2. Glue2 终止 TLS 连接并读取 
    1Host
     标头。
  3. Glue2 检查身份验证（会话 cookie、API 密钥或 OAuth 令牌）。如果端点需要身份验证而未提供，则请求被拒绝并返回 
    1401 Unauthorized
    。
  4. Glue2 将虚拟主机解析为项目 ID 和后端服务目标。
  5. Glue2 将身份验证和项目标头注入请求。
  6. 请求被转发到后端服务。
  7. 后端响应被流式传回客户端，Glue2 记录事务。

注入的标头

转发到后端时，Glue2 注入：

标头 描述 
1X-Glue-Authentication
 Base64 编码的 JWT，包含用户 ID、会话信息、角色和身份验证方法 
1X-Holacloud-Project-Id
 从虚拟主机派生的项目 UUID 
1X-Holacloud-Tenant-Project-Id
 用于多租户隔离的租户范围项目 ID 
1X-Forwarded-For
 原始客户端 IP 地址 
1X-Forwarded-Proto
 原始协议（
1http
 或 
1https
）

后端服务使用这些标头识别调用者并执行授权。

1X-Glue-Authentication
JWT 使用仅 Glue2 和受信任后端已知的共享密钥进行签名。

负载均衡与故障转移

Glue2 将流量分布到每个服务的多个后端实例。如果某个实例不健康（健康检查失败或返回错误），Glue2 将其从轮换中移除并在健康实例上重试。健康检查每 10 秒执行一次。您可以通过 

1/v0/status
端点监控后端健康状态。
